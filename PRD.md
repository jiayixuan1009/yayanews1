## 零、第一性原理：什么决定了「最快」

> **本段概要**：新闻从「世界上发生」到「读者在手机里看到」中间有好几步。我们把每一步能省下的时间加起来，就是能不能比竞品快。这里用一张简单链条说明：**哪几步我们能动刀**，为什么不能只盯某一个环节。

一条金融新闻从发生到被中文用户阅读，经过以下不可跳过的环节：

```
事件发生 → 信源发布 → 我方感知 → 翻译/加工 → 入库 → 用户可见
   T0         T1          T2          T3        T4       T5
```

**总延迟 = T5 − T0 = (T1−T0) + (T2−T1) + (T3−T2) + (T4−T3) + (T5−T4)**

| 环节 | 变量 | 我们能控制什么 |
|------|------|----------------|
| T1−T0：信源反应时间 | 取决于 API/信源的速度 | 选最快的信源，多源竞速 |
| T2−T1：我方感知延迟 | 轮询频率 / WebSocket / Webhook | 提高采集频率，用推送替代轮询 |
| T3−T2：加工延迟 | LLM 翻译/改写时间 | 并行化、选更快的模型、精简 prompt |
| T4−T3：入库延迟 | 数据库写入 | 可忽略（毫秒级） |
| T5−T4：用户可见延迟 | 页面缓存/ISR/轮询间隔 | 缩短 revalidate、用 WebSocket 推送 |

**结论：要做到最快，每个环节都必须以秒为单位优化，而不是以分钟。**

## 二、当前系统架构与实测延迟

> **本段概要**：这一章回答两件事：**网站是用什么技术搭的**，以及**一条快讯、一篇文章从采集到用户看到大概要多久**。数字会随优化变化，重点是理解「时间花在哪」和「AI 调用越多越贵越慢」。

### 2.1 技术栈

> **本段概要**：像「菜单」一样列出：前台用什么框架、数据存在哪、内容谁自动生成、快讯从哪些渠道来。不必记英文名，知道「多层配合」即可。

| 层 | 技术 | 选型理由 |
|----|------|----------|
| **前端** | Next.js 14 (App Router) + TypeScript + Tailwind CSS | SSG/ISR/SSR 混合渲染，SEO 友好 |
| **数据库** | SQLite (better-sqlite3, WAL) | 零运维，单文件，读写不阻塞，满足当前吞吐 |
| **内容 Pipeline** | Python 多 Agent：**采集→生成→审核→SEO→发布**；选题/写作/SEO 元数据 **多路并行**；**写作可同次产出 SEO 字段，Agent4 完整时跳过元数据 LLM**，仍保留 slug/内链/免责声明/CTA | 阶段逻辑不变，批次总耗时与 token 可降 |
| **LLM** | DeepSeek V3 (deepseek-chat) / R1 (deepseek-reasoner) | 智能分发：常规快讯与文章走高并发V3，深度解析走满血R1模型保证逻辑深度。成本仅为GPT-4十分之一 |
| **快讯采集** | 多通道：Finnhub REST、**新浪财经滚动 + 中文 RSS**、Marketaux、CryptoCompare、CoinGecko、英文 RSS、**Finnhub WS→JSONL 缓冲（周期 run 批量翻译入库）**、LLM 兜底 | 中文源减英译；WS 缩短信源到缓冲的等待（相对纯轮询） |
| **常驻与部署** | **PM2**：`finnhub_ws_flash` + `run_daemon`（间隔见 **§十五.1**） | 7×24 感知与生产 |
| **用户侧刷新** | **SSE**：`/api/flash/events`、`/api/live/events`（服务端轮询 DB 推事件，前端再拉 JSON） | 新快讯/新文露出加速；与「真 WS 广播」区别见 **§十五.3** |

### 2.2 当前实测延迟链（从信源发布到用户可见）

> **本段概要**：**快讯**像短消息：**文章**像长稿。表里是「理想情况下大概多少秒/分钟」——英文要多一步翻译就会慢；用户浏览器隔十几秒才刷新一次列表，也会让人感觉晚一点才刷出来。

#### 快讯链路

| 环节 | 当前耗时 | 瓶颈原因 |
|------|---------|---------|
| 通道采集（多路并发，含中文源） | ~12–18 秒 | 通道数增加；中文条目**免翻译**，英文仍走批量翻译 |
| 英文翻译（批量 LLM） | ~15–50 秒 | **中文比例↑则批次数↓**；仍受英文条数制约 |
| 分类 + 去重 + 入库 | ~1 秒 | 无瓶颈 |
| **Pipeline 小计** | **~30–55 秒**（英文多时偏高） | 中文源+WS 缓冲有助于压低翻译与感知等待 |
| 用户可见（客户端轮询） | 0–15 秒 | 快讯页 **15s** 轮询（壳层策略见 §3.1） |
| **端到端** | **~45 秒–1.5 分钟**（典型） | **目标: &lt; 30 秒**（继续推 SSE/更密采集） |

#### 文章链路

| 环节 | 当前耗时 | 瓶颈原因 |
|------|---------|---------|
| Agent 1：选题采集（4 次 LLM 串行 + RSS） | ~25–70 秒 | 4 个分类串行 LLM 调用 |
| Agent 2：内容生成（并行，**含 SEO 字段同次输出**） | ~100–300 秒 | 单篇略增 token；**省去 Agent4 元数据 LLM 时整体批次缩短** |
| Agent 3：质量审核 | ~0 秒 | 纯规则，无 LLM |
| Agent 4：SEO 后处理 | **~0–80 秒** | **完整写作 SEO 时仅规则+slug，无元数据 LLM**；不全时仍逐篇补 SEO LLM |
| Agent 5：入库发布 + Google Ping | ~2 秒 | 无瓶颈 |
| **Pipeline 小计** | **~3–9 分钟（10 篇）** | |
| 用户可见（ISR 缓存） | 0–60 秒 | 首页/列表 revalidate=60s |
| **端到端（单篇）** | **~1–2 分钟** | **目标: &lt; 3 分钟** |

### 2.3 LLM 调用清单（成本和速度的核心）

> **本段概要**：**LLM** 可以理解成「很贵的智能写作/翻译服务」。每跑一轮要调用多少次，大致就等于要花多少时间和预算。合并写作与 SEO、增加中文快讯，都是为了**少调用几次**。

| 调用点 | 次数/轮 | 方式 | max_tokens | 可否省略 |
|--------|---------|------|-----------|---------|
| 选题生成 | 4 | 串行 | 4096 | 有 RSS 素材时可省 |
| 文章写作 | 10 | 并行 | 4096 | **已合并 SEO 元数据到同次输出（成功则省下方一行）** |
| SEO 元数据 | **0–10** | 并行 | 800 | **写作 JSON 完整时跳过；否则仍调用** |
| 快讯翻译 | N/8 批 | 串行 | 3000 | 不可省（英文源） |
| LLM 兜底 | 0–1 | — | 2000 | 仅兜底 |
| **合计** | ~25–30 次 | | | |

### 2.4 采编到发布时效（产品口径与数据闭环）

> **本段概要**：公司内说「这条有多快」，要统一口径：**从我们开始采集/选题的时间点，到网站上能点开的时间点**，中间差多少秒。后台列表和报表都按这个算，方便运营和研发对表。

**定义**：「获取」指 Pipeline 记录的信源感知或选题落库时间（`collected_at`）；「发布」指文章/快讯在库中可对外展示的时间（`published_at`）。**采编到发布耗时** = 两者之差，与后台展示的 `processing_seconds` 一致。

| 类型 | 产品目标（方向） | 当前量级（见 2.2） | 数据落点 |
|------|------------------|-------------------|----------|
| **快讯** | 越短越好，目标向 **&lt; 30s～1min** 收敛 | ~1–2min（含轮询） | `flash_news.collected_at` → `published_at` + Admin 列表 |
| **文章** | 深度可放宽，标准资讯向 **&lt; 3～5min/篇** 收敛（批次另计） | 单篇约 1–2min + ISR 最多 60s | `articles.collected_at` → `published_at` + 速度监控 `pipeline_runs` |

**对内工具**：Admin「速度监控」看 Pipeline 各阶段与通道耗时；「时效对比」用 Google News 等结果估算与竞品首发的时间差。路线图中的并行化、WebSocket 信源、推送替代轮询均直接服务于缩短上表时间。

## 三、前端系统

> **本段概要**：**前端**就是读者在浏览器里看到的页面：首页、资讯列表、文章、快讯、行情等。这里说明每个页面大概多久会「自动换一批新内容」，以及我们为**谷歌收录**和**打开速度**做了哪些页面级设计。

### 3.1 页面与渲染策略

> **本段概要**：不同页面更新频率不一样：有的每分钟换一次缓存，有的用户一打开就现查数据库。**revalidate** 可以理解成「最多隔多少秒页面算过期、要重新生成」。

| 页面 | 渲染 | revalidate | 用户延迟 | 核心功能 |
|------|------|-----------|---------|---------|
| 首页 `/` | ISR | 60s | 0–60s | 分类入口 + 最新文章 + 实时 Ticker + 快讯 + 热门标签 |
| 资讯列表 `/news` | ISR | 60s | 0–60s | 全部/分类筛选 + 深度 Tab + 分页 |
| 分类页 `/news/[category]` | SSG+ISR | 60s | 0–60s | 4 大分类预生成；衍生品含子分类 Tab |
| 文章详情 `/article/[slug]` | 动态 SSR | 300s | — | 正文 + 阅读进度 + 分享 + 上下篇 + JSON-LD |
| 快讯 `/flash` | 动态+CSR | 30s(壳) | **~15s(列表轮询)** | 按日期分组 + 重要度 + 倒计时刷新 |
| 行情 `/markets` | CSR | 60s | — | CoinGecko 实时行情 + 7D 走势 |
| 搜索 `/search` | 动态 | — | 即时 | 全文搜索 + 热门标签，noindex |
| 专题/指南 | SSG | 300–600s | — | 低频更新内容 |
| Admin `/admin/*` | CSR | — | — | 后台管理，noindex |

### 3.2 公共组件

> **本段概要**：全站共用的「积木」——顶栏导航、底栏、行情条、文章卡片、分享按钮等。知道有这些模块即可，方便和设计、运营对齐文案与入口位置。

| 组件 | 功能 | 技术要点 |
|------|------|----------|
| **Header** | 全局导航 + 下拉菜单 + 搜索 + CTA | aria 无障碍，全站内链 |
| **Footer** | 品牌 + 分类索引 + 工具链接 | SEO 内链矩阵 |
| **LiveTicker** | BTC/ETH/SOL/XRP 实时行情 | CoinGecko API，30s 刷新，dynamic import |
| **ArticleCard** | 文章卡片 (featured/normal) | next/image AVIF/WebP（允许域名内优化） |
| **ReadingProgress** | 阅读进度条 | rAF 节流 |
| **ShareButtons** | 社交分享 | Toast 替代 alert |
| **CtaBanner** | 交易引导横幅 | 每个内容页底部 |

### 3.3 页面加载速度与 PageSpeed Insights（优秀站点标准）

> **本段概要**：谷歌会给网站打分（类似「手机打开快不快」）。分数高有利于搜索排名和用户体验。这里写我们**朝哪些指标努力**，以及常用优化手段（压图、缓存等）。

**Google PageSpeed Insights / Lighthouse 常见「优秀」参考**（移动端为主）：Performance 评分 **≥ 90**；Core Web Vitals：**LCP ≤ 2.5s**、**INP ≤ 200ms**、**CLS ≤ 0.1**。

| PSI 关注点 | 本站实现 | 与「优秀」的关系 |
|------------|----------|------------------|
| 图片体积与格式 | `next/image` + AVIF/WebP；外链 CDN 白名单内可走优化 | 降低 LCP、节省带宽 |
| 懒加载 | 非首屏图默认 lazy；行情小图标 lazy + async decode | 减少首屏竞争 |
| 压缩 | `compress: true`；生产构建压缩 JS/CSS | 传输更小 |
| 阻塞渲染 | `next/font` 构建期自托管 Inter；LiveTicker 等 dynamic | 减少阻塞 |
| 缓存 | `/_next/static` 长期缓存；图片与 sitemap 分层缓存 | 回访与 CDN 友好 |
| TTFB | SSR/ISR 依赖 SQLite 与 Node；高并发时需 CDN + 查库优化（见第八章） | 目标 **&lt; 600ms** |

**验收**：上线后以 PageSpeed Insights 对首页、资讯列表、文章详情、快讯页各跑 1 次；未达 90 时按第八章逐项排期。

### 3.4 各级页面与谷歌新闻类站点最佳实践（页面层）

> **本段概要**：谷歌希望新闻站有：**稳定链接、能看清发布时间、手机能用、地图（sitemap）齐全**。我们对照一张自查表，方便 SEO 与产品验收。

以下对齐 Google 对**新闻/资讯类网站**的通用建议，按页面类型落表。

| 页面/模块 | 谷歌侧期望 | 本站实现 |
|-----------|------------|----------|
| **全站** | HTTPS、可抓取、移动端可用 | HTTPS；`robots.txt` 放行内容、屏蔽 `/admin` `/api`；响应式 |
| **首页** | 清晰主题、主导航、新鲜内容入口 | 分类入口、最新文章/快讯、内链 |
| **列表** | 稳定 URL、Tab/分页可抓取 | ISR/SSG；深度 Tab；规范链接 |
| **文章** | 独立 URL、标题正文、**发布时间/修改时间**、作者或机构 | `NewsArticle` JSON-LD；`published_at`/`updated_at`；来源标注 |
| **快讯** | 短讯可索引则需时间与来源 | 时间线；重要度；采集/发布时间可追溯 |
| **专题/指南** | 聚合页与子页关系清楚 | 专题列表与详情；指南可扩展 HowTo |
| **站内搜索** | 可不索引但需可用 | noindex |
| **Sitemap** | 新闻站提交 news sitemap + 常规 sitemap | `sitemap.xml` + `sitemap-news.xml`；发布后 Google Ping |

**内容侧**：真实时间、来源、风险提示与 Pipeline 改写/审核策略一致。

## 四、后端 Pipeline

> **本段概要**：**后端**是在服务器上自动干活的那套程序：找选题、写文章、检查质量、加 SEO、发上网；快讯则是多路抓新闻、翻译、入库。这里也写了**为什么**要加中文源、WebSocket、常驻进程、以及把写作和 SEO 合成一步。

### 4.1 文章流水线（5 Agent）

> **本段概要**：可以想象成 **5 个工位**：选题 → 写稿 → 质检 → 加链接和合规段落 → 发布。前两个最费 AI；我们现在尽量在「写稿」一步里多交一些 SEO 信息，后面就少再问一遍 AI。

```
Agent 1 选题采集 → Agent 2 内容生成（可含 SEO 元数据） → Agent 3 质量审核 → Agent 4 SEO 后处理 → Agent 5 发布
```

| Agent | LLM? | 关键点 |
|-------|------|--------|
| **1 选题** | 是 | 配额 + 标题去重 |
| **2 写作** | 是 | standard / deep；**同次 JSON 输出** `content` + `seo_title` / `seo_description` / `tags` / `sentiment` / `tickers` / `key_points` |
| **3 审核** | 否 | 字数/HTML/去重/摘要（见下 **摘要口径**） |
| **4 SEO** | **条件** | **写作侧字段完整 → 不调元数据 LLM**，仍执行：slug 唯一、正文内链、免责声明、Yayapay CTA；不完整则 **补一轮 SEO LLM**（与历史行为一致） |
| **5 发布** | 否 | SQLite + Google Ping |

**为何合并写作+SEO 元数据**：文章链路中「写作 + SEO 元数据」原占两轮 LLM，批次耗时长、调用次数多。**解决**：在质量可控前提下合并为一次结构化输出，**达成**：单轮成功时 **减少约 10 次/批 SEO 元数据调用**，缩短文章 Pipeline 墙钟，成本下降；失败或字段不全时 **自动回退** Agent4 LLM，不破坏原有兜底逻辑。

### 4.2 快讯引擎

> **本段概要**：快讯要从很多网站/API 拉标题和摘要。**中文源**来了就能直接发，省翻译时间；**WebSocket** 像「推送订阅」，有新消息先堆在缓冲区，定时一起处理。多条路并行，一条断了别的还能用。

**为何扩展中文源与 WS**：英文源必经「批量翻译」，拉长 T3；纯 REST 轮询拉长 T2。**解决**：① **cn_sina**（新浪财经滚动 API）与 **cn_rss**（可配置中文财经 RSS）直连中文，**跳过英译**；② **Finnhub WebSocket** 独立进程推送新闻至 `data/ws_flash_queue.jsonl`，每次 `collect_flash` **优先 drain 缓冲**再与 REST 结果合并翻译入库。**达成**：同样条数下 **翻译批次数下降**；有 WS 守护时 **信源侧更接近实时推送**（入库仍受整轮 collect 频率约束）。

**通道与权重（摘要）**：中文源权重较高以便优先入库；REST Finnhub 与 WS 条目均为英文时需翻译。健康检查：连续失败自动降级（含新增通道）。

### 4.3 常驻进程与 PM2（运维落地）

> **本段概要**：如果只靠人点按钮或一天只跑一两次，网站上会长时间没有新内容。**常驻**就是让服务器里有两个长期开着的程序：一个专门听新闻推送，一个按固定间隔自动跑「抓快讯 / 发文章」。**PM2** 是常用的进程管家，崩了会自动拉起。

**为何做**：仅依赖人工或稀疏 cron 时，快讯与文章生产存在 **长时间空窗**，与「最快首发」目标冲突。**解决**：① `python -m pipeline.daemon.finnhub_ws_flash`（需 `FINNHUB_KEY`）持续写 WS 缓冲；② `python -m pipeline.run_daemon` 按环境变量周期执行 `--flash-only` / `--articles-only`（如 `DAEMON_FLASH_SEC`、`DAEMON_ARTICLE_SEC`）。**达成**：**7×24 信源感知 + 周期入库/发文**，可与 PM2 `ecosystem.config.cjs`（`yaya-finnhub-ws` + `yaya-pipeline-daemon`）同机部署。**间隔与成本怎么权衡**：见 **§十五.1**。**间隔与成本权衡的定稿方案见 §十五.1**。

### 4.4 数据库设计

> **本段概要**：内容、标签、每次跑批花了多久、和竞品比快还是慢，都存在**数据库表**里。非技术同事只需知道：**要查产量、耗时、时效对比，都有表可查**。

| 表 | 关键字段 | 用途 |
|----|----------|------|
| **articles** | collected_at, published_at, processing_seconds | 正文内容 |
| **flash_news** | 同上 + importance | 快讯 |
| **categories / tags / article_tags** | — | 分类与长尾 |
| **topics / guides** | — | 专题与教程 |
| **pipeline_runs** | 阶段与通道耗时 JSON | 速度分析 |
| **speed_benchmarks** | 竞品首发时间差 | 时效对比 |

**摘要口径（写作+SEO 合并后）**：`articles.summary` 在写作 JSON 完整时 **与 SEO 描述（`seo_description`）对齐**，列表卡片与 OG 描述共用该字段——**偏收录与点击率**；若需「中性短摘要」与 Meta 分离，可另立字段排期（见 §十四）。

**PSI 归档表**：`psi_reports`（PageSpeed 跑分历史），由 `python -m pipeline.tools.run_psi` 写入；迁移 `python -m pipeline.migrate_psi_reports`。详见 **§14.5**。

### 4.5 外部 API

> **本段概要**：我们要向多家外部服务「买数据或买算力」：行情、新闻、AI 写作、通知谷歌更新站点地图等。谁贵谁慢，和 §2 的耗时表一起看。

MiniMax、Finnhub（REST + **可选 WS**）、Marketaux、CryptoCompare、CoinGecko、RSS（含中文源 URL 配置）、Google Ping — 成本与延迟见 §2；WS 依赖 `websocket-client` 与 `FINNHUB_KEY`。

## 十、改进路线图：如何做到「全世界最快」

> **本段概要**：分四阶段从「现在能用」走到「和头部竞品比速度」。已打 ✅ 的是已经做掉或部分做掉的，其余是排队中的能力，不是承诺完成日期。

**第一阶段（持续推进）**：Agent 并行、快译并发、文章/快讯解耦、LLM 超时 — 目标快讯 Pipeline 向 20–35s 收敛（随中文占比↑），文章批次向 2–4min 收敛（随 SEO 合并成功率↑）。

**第二阶段（部分已落地）**：✅ **中文快讯源**、✅ **Finnhub WS→缓冲**；待：**NewsAPI**、垂直 API、Twitter/SEC、Webhook、全量推送替代轮询。

**第三阶段（部分已落地）**：✅ **PM2 常驻（WS + run_daemon）**；待：面向用户的 **SSE/WS 推送**、Redis 队列、更快 LLM、流式翻译、CDN、PostgreSQL NOTIFY、多地域。

**第四阶段**：智能去重聚合、重要性评分、个性化、图表、多语言。

## 十四、可视化与可验证指标体系

> **本段概要**：**不能只说感觉快**，要有数字：每条处理多久、每轮 Pipeline 多久、和竞品谁更早、网页分多少。这里定义「看什么数、在哪看、例会怎么复盘」，方便管理层抽查。

> **本段概要**：证据分三类：能算、能对比、能画图——避免口水战。

| 类型 | 含义 |
|------|------|
| **可验证** | 每条数据有库表或日志可复算 |
| **可比较** | 同周期 A vs B（优化前后、我方 vs 竞品） |
| **可视化** | 图表/分布/趋势，非仅表格 |

### 14.2 指标字典（例会看板）

> **本段概要**：指标「词典」：每个名词什么意思、数据从哪来。做报表或拉会时大家对齐用。

| 指标 | 定义 | 来源 | 展示 | 比较方式 |
|------|------|------|------|----------|

### 14.5 运维定时任务与环境变量

> **本段概要**：PSI 与周报需要服务器定时执行；变量写在系统环境或 PM2 `env` 里，**不要提交密钥到 Git**。

| 变量 | 用途 | 示例 |

| `DAEMON_FLASH_SEC` / `DAEMON_ARTICLE_SEC` | `run_daemon` 周期间隔（秒） | `300` / `7200` |

| `SMTP_HOST` `SMTP_PORT` `SMTP_USER` `SMTP_PASSWORD` | 发信 | 企业邮箱或 SendGrid 等 |
| `DIGEST_TO_EMAIL` `DIGEST_FROM_EMAIL` | 周报收件人；发件人 | 多个收件人用英文逗号 |
| `LLM_API_KEY` `FINNHUB_KEY` 等 | Pipeline / WS 守护 | 与线上一致 |

**Cron 示例（Linux，路径改为实际 `biyanews` 目录，前置 `export` 或 dotenv）**

```bash
# 每周一 06:00 UTC 发 Pipeline 周报
0 6 * * 1 cd /opt/biyanews && . ./.env && python -m pipeline.tools.weekly_digest_email
```

## 十五、未落地项技术方案（PM2 间隔与 7×24 · PG + Redis · 真 WS 广播）

> **本段概要**：下面三块**尚未开发**，是先写清「为什么要做、做成什么样、什么时候做」供你拍板。与 **§十六**「换工具复盘」互补：§十六偏选型对比，§十五偏**可执行方案**。

### 15.1 PM2 间隔策略与 7×24 常驻

**为什么要做**

| 痛点 | 说明 |
|------|------|
| **首发空窗** | 仅靠人工点 Admin 或一天一两次 cron，用户在数小时级看不到新快讯/新文，与「最快」定位冲突。 |
| **间隔两难** | 间隔**过短** → 外部 API / LLM 调用暴涨，费用与限流风险上升，单机 CPU 打满。间隔**过长** → 即使用上 WS 缓冲，**入库节奏**仍慢，用户侧仍觉得「慢半拍」。 |
| **职责分离** | **Finnhub WS 进程**负责「尽量不漏信源」；**run_daemon** 负责「按可控成本批量翻译与入库」；两者拆开才能分别调参。 |

**方案目标**

- 在**公司可接受的 API+LLM 月成本**内，把「站内可见新快讯」的中位延迟压到目标区间（参见 §十一）。
- 文章生产与快讯**解耦**：文章可 1～4 小时一轮，快讯 3～15 分钟一轮（按环境分级）。

**推荐参数带（非强制，运维可改 env）**

| 环境 | `DAEMON_FLASH_SEC` | `DAEMON_ARTICLE_SEC` | 说明 |
|------|---------------------|----------------------|------|
| 验证/低成本 | 600～900 | 14400～28800 | 先跑稳再加密 |
| 生产默认 | 180～600 | 3600～7200 | 与当前 PRD 量级一致 |
| 竞速/活动期 | 120～180 | 1800～3600 | 需同步监控费用与错误率 |

**需要你判断**

- 每月 **Pipeline+信源** 费用上限 → 反推最快可接受的 `DAEMON_FLASH_SEC`。
- 是否 **7×24** 全开 WS + daemon，还是仅交易时段开（省夜间调用）。

### 15.2 PostgreSQL + Redis + 真正 WebSocket 广播

**为什么要做（与现状对比）**

| 现状 | 瓶颈 |
|------|------|
| 用户侧 **SSE**（`/api/flash/events`、`/api/live/events`） | 每个在线用户一条长连接，服务端**周期性查库**（`MAX(id)`）。连接数上来后，**SQLite 锁竞争**、Node 进程 FD 与 CPU 占用上升。 |
| Python Pipeline 写库、Next 读库 | 同盘 SQLite **多读一写**在高峰下延迟抖动。 |
| 无中央消息总线 | 无法做「写一次、千万端同时推」的**真广播**；多实例 Next 部署时，每台各自轮询，放大读压力。 |

**为什么要做（目标能力）**

1. **PostgreSQL**：主库存文章/快讯；支持**多 Next 实例**并发读、复杂报表；通过 **`LISTEN/NOTIFY`** 或逻辑日志，在**有新行时**通知中间层，替代「每 2～3 秒全员扫库」。
2. **Redis**：  
   - **队列**：`collect_flash` / `run_article_pipeline` 任务入队，多 Worker 消费，**削峰、重试、可视长度**。  
   - **Pub/Sub**：新快讯 id 或摘要 payload 发布到频道 `flash:new`，由 **WS 网关**转发给订阅用户。
3. **真正 WebSocket 广播**：独立 **WS 服务**（或 Next 外挂 Socket.IO 等），浏览器 `wss://` 订阅；网关只维护**内存订阅表**，不写业务库读路径，延迟毫秒级。

**分期落地（降低一次性迁移风险）**

| 阶段 | 内容 | 触发建议 |
|------|------|----------|
| **P-α** | PG 迁移 + 双写/只读切换方案 | 单机 QPS 或 Admin 报表需求压不住 |
| **P-β** | Redis 队列，Pipeline worker 与 Web 解耦 | 跑批堆积、LLM 超时频发 |
| **P-γ** | NOTIFY → WS 网关 → 浏览器 | **同时在线 SSE/WS 连接 > 500**（或 P95 TTFB 超线）再立项 |

**需要你判断**

- **何时切 PG**：是否以「第二台应用机」或「读放大」为硬触发条件。  
- **WS 网关自建 vs 云服务**（如 Ably/Pusher）：自建省月费、运维成本高；云服务相反。

## 十六、方案复盘：若此处改用 XX 会更好（供内部讨论）

> **本段概要**：用大白话说：**如果现在换成别的数据库、别的任务调度、别的新闻源，会不会更好**。表里的结论是「以后规模大了可以再换」，不是「现在做错了」。给架构评审或非技术老板一个「为什么没一步到位」的交代。

> 以下为**非当前承诺**，仅表达「若重来或升级时值得考虑的替换」——方便评审会问「为什么不直接上 XX」时一页说清。

| 现状取舍 | 若这么改 / 换成 XX | 可能更好在哪 | 为何当前没先上 |
|----------|---------------------|--------------|----------------|
| **SQLite 单文件** | **PostgreSQL**（或 Neon/Supabase 托管 PG） | 多实例写、LISTEN/NOTIFY 推前端、复杂查询与报表、连接池 | 零运维、单机足够；迁移与双写有成本；**落地方案见 §十五.2** |
| **PM2 + 子进程跑 Python** | **Kubernetes Job / Cloud Run / AWS Lambda 定时** | 弹性扩缩、故障隔离、与云监控一体 | 团队熟悉度与冷启动；当前量级 cron 够用 |
| **Finnhub REST + WS** | **Polygon、NewsAPI、Benzinga、交易所官方 WS** | 标的/市场更贴业务、延迟或覆盖更优 | 成本、合规、接入量；Finnhub 综合门槛低 |
| **自研 ThreadPool 采集** | **Celery / RQ + Redis** | 任务队列、重试、可视监控、削峰 | 多一个常驻组件；当前 Python 脚本链简单直接 |
| **DeepSeek (V3/R1)** | **多模型多模态矩阵**（GPT-4o/Claude 3.5） | 针对跨模态图文生文、复杂财报 PDF 解析更强 | 目前文本新闻解析中 DeepSeek 兼顾极速与极低成本，效果极佳 |
| **批量 LLM 翻译快讯** | **流式翻译或专用翻译 API** | T3 更短 | 质量与格式控制；批量 JSON 更稳 |
| **Next.js 自托管 SQLite** | **Vercel + 外置 DB + Edge** | 全球边缘、TTFB | 费用与数据层改造 |
| **手工/半自动 PSI** | **Lighthouse CI / PageSpeed API 定时入库** | §14.4 的 PSI 自动闭环 | 需 API Key 与存储设计 |
| **前端轮询快讯** | **SSE 或 Ably/Pusher/自建 WS** | 用户侧「最快」感知 | 连接数、鉴权与降本 |

**一句话**：当前方案优先 **少组件、快上线、可度量**；上表各项多在 **规模、多活、极致延迟** 阶段再换工具更划算。若公司统一技术栈已有 **K8s / PG / Redis**，可优先评估队列与库两件。
